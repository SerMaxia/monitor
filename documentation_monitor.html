Vue d'ensemble

Le script monitor.sh est un outil de monitoring système automatisé en Bash. Il effectue une série de vérifications sur une machine Linux et génère un ensemble de rapports dans un dossier horodaté. Chaque exécution produit un nouveau dossier de rapport, permettant un suivi historique de l'état du système.

ℹ️ Objectif principal

Collecter, analyser et stocker des informations clés sur l'état du système : services actifs, logs d'erreurs, processus consommateurs de CPU, anomalies dans les fichiers de logs et fichiers sensibles.

Flux d'exécution

Initialisation

Affichage du contexte (heure, machine, utilisateur) et création du dossier de rapport.

Collecte des données

Vérification des services, lecture des logs journalctl, observation des processus.

Analyse

Analyse de app.log et metrics.csv, recherche de fichiers .bak et de secrets.

Vérification finale

Contrôle que tous les fichiers attendus ont bien été générés.

Configuration & Variables

Le script utilise plusieurs variables de configuration définies en début de fichier pour déterminer les chemins de travail.

monitor.sh — lignes 3 à 13 Bash 
# Configuration
USER_NAME=${USER:-$(whoami)}
BASE_DIR="/opt/monitoring-lab/$USER_NAME"
REPORT_DIR="$BASE_DIR/reports/run-$(date +%Y%m%d-%H%M%S)"
LATEST_LINK="$BASE_DIR/reports/latest"
LOG_DATA_DIR="/var/log/monitoring-lab/data"
TREE_DIR="/var/log/monitoring-lab/tmp/tree"

# Variables pour l'affichage du contexte
START_TIME=$(date)
MACHINE_NAME=$(hostname)
Variable Description Exemple de valeur
USER_NAME Nom de l'utilisateur courant maxia
BASE_DIR Dossier racine du monitoring pour l'utilisateur /opt/monitoring-lab/maxia
REPORT_DIR Dossier du rapport courant (horodaté) .../reports/run-20260625-093000
LATEST_LINK Lien symbolique vers le dernier rapport .../reports/latest
LOG_DATA_DIR Répertoire des fichiers de données (app.log, metrics.csv) /var/log/monitoring-lab/data
TREE_DIR Répertoire cible pour la recherche find/xargs /var/log/monitoring-lab/tmp/tree
⚠️ Attention — Exécution avec sudo

Si le script est lancé avec sudo, la variable $USER sera root. Les rapports seront alors créés dans /opt/monitoring-lab/root/ au lieu du dossier de l'utilisateur.

Étape 1 — Mise en place des dossiers rapports

Le script commence par créer le dossier de rapport horodaté et un lien symbolique latest qui pointe toujours vers le rapport le plus récent.

Création des dossiers Bash 
mkdir -p "$REPORT_DIR"
ln -sfn "$REPORT_DIR" "$LATEST_LINK"
cd "$REPORT_DIR" || exit 1
CommandeRôle
mkdir -p Crée le dossier (et ses parents) s'il n'existe pas encore
ln -sfn Crée ou met à jour le lien symbolique latest (-s symbolique, -f force, -n no-dereference)
cd ... || exit 1 Se déplace dans le dossier de rapport ; quitte avec erreur si impossible

Étape 3 — Lecture des logs (journalctl)

Le script extrait les logs récents et les erreurs des services via journalctl. Deux fichiers sont produits :

FichierContenuCommande clé
journald.txt Les 30 dernières lignes de logs de fake-api et log-generator journalctl -u <service> -n 30
journald_errors.txt Uniquement les erreurs (priorité err à emerg) des deux services journalctl -u ... -p err..emerg
ℹ️ Niveaux de priorité syslog

Le filtre -p err..emerg capture les niveaux : err (3), crit (2), alert (1) et emerg (0) — c'est-à-dire toutes les erreurs graves.

Étape 4 — Observation des processus

Le script capture un instantané des processus les plus gourmands en CPU.

Commandes exécutées Bash 
# Top 10 des processus par utilisation CPU
ps -eo pid,ppid,cmd,%mem,%cpu --sort=-%cpu | head -n 11 > top_cpu.txt

# Recherche spécifique du processus cpu-hog.sh
ps aux | grep "[c]pu-hog.sh" > hogs.txt
FichierContenu
top_cpu.txt Les 10 processus les plus gourmands en CPU (PID, PPID, commande, %MEM, %CPU)
hogs.txt Les processus correspondant au script cpu-hog.sh (processus volontairement gourmands)
💡 Astuce — Le grep avec crochets

L'utilisation de [c]pu-hog.sh au lieu de cpu-hog.sh est une astuce classique : elle empêche grep de se trouver lui-même dans les résultats, car la commande grep "[c]pu-hog.sh" ne correspond pas au motif recherché.

Étape 5 — Analyse de app.log

Le script analyse le fichier /var/log/monitoring-lab/data/app.log pour en extraire des statistiques et produire une version nettoyée.

Statistiques extraites

MétriqueCommandeDestination
Nombre total de lignes wc -l < app.log app_summary.txt
Nombre de lignes ERROR grep -c "ERROR" app.log
Nombre de lignes CRITICAL grep -c "CRITICAL" app.log
Top 5 des hôtes awk '{print $1}' | sort | uniq -c | sort -nr | head -n 5

Rédaction des secrets

Masquage des informations sensibles Bash 
sed -E 's/SECRET=[a-zA-Z0-9]+/SECRET=**REDACTED**/g' "$APP_LOG" > app_redacted.log
🔒 Sécurité — Rédaction automatique

Toutes les occurrences de SECRET=<valeur> dans le fichier de logs sont remplacées par SECRET=**REDACTED**. Cela permet de partager les logs sans exposer de données sensibles.

Étape 6 — Analyse de metrics.csv

Le script analyse le fichier CSV de métriques pour identifier les lignes où l'utilisation CPU dépasse 95%.

Extraction des anomalies CPU Bash 
# Format CSV supposé : id,host,cpu_pct,...
M_HIGH_CPU=$(awk -F',' '$3 >= 95 {print}' "$METRICS_CSV" | wc -l)
M_ANOMALIES=$(awk -F',' '$3 >= 95 {print}' "$METRICS_CSV" | head -n 5)

Le fichier metrics_summary.txt contient :

  • Le nombre total de lignes dans le CSV
  • Le nombre de lignes où cpu_pct ≥ 95%
  • Les 5 premières lignes considérées anormales

Étape 7 — Recherche avec find & xargs

Le script explore l'arborescence /var/log/monitoring-lab/tmp/tree pour rechercher des fichiers spécifiques.

Commandes find & xargs Bash 
# Trouver tous les fichiers .bak
find "$TREE_DIR" -type f -name "*.bak" > tree_bak.txt

# Trouver les fichiers contenant "SECRET="
find "$TREE_DIR" -type f -print0 | xargs -0 grep -l "SECRET=" > tree_secret.txt
FichierContenuDétails
tree_bak.txt Liste des fichiers .bak Fichiers de sauvegarde potentiellement obsolètes
tree_secret.txt Liste des fichiers contenant SECRET= Fichiers contenant potentiellement des données sensibles
ℹ️ Pourquoi -print0 et -0 ?

L'option -print0 de find et -0 de xargs utilisent le caractère nul (\0) comme séparateur. Cela garantit un fonctionnement correct même avec des noms de fichiers contenant des espaces ou des caractères spéciaux.

Étape 8 — Contrôle final

Le script effectue une vérification finale pour s'assurer que tous les fichiers attendus ont bien été générés.

Liste des fichiers vérifiés Bash 
FILES_TO_CHECK=("services.txt" "journald.txt" "journald_errors.txt"
    "top_cpu.txt" "hogs.txt" "app_summary.txt"
    "app_redacted.log" "metrics_summary.txt"
    "tree_bak.txt" "tree_secret.txt")

Pour chaque fichier manquant, un message d'erreur est affiché. Le résultat final sera :

  • Tous présents : Contrôle terminé : Tous les fichiers requis sont présents.
  • Fichier(s) manquant(s) : Contrôle terminé : Il manque X fichier(s).
⚠️ Note importante

Certains fichiers (comme app_summary.txt, metrics_summary.txt, tree_bak.txt, tree_secret.txt) ne sont générés que si les fichiers/dossiers sources existent (app.log, metrics.csv, tree/). Le contrôle final signalera ces fichiers comme manquants si les sources n'étaient pas présentes.

Utilisation

Lancer le script

Terminal Linux Shell 
# Rendre le script exécutable (une seule fois)
chmod +x monitor.sh

# Exécuter le script
sudo ./monitor.sh

Consulter les résultats

Accéder au dernier rapport Shell 
# Via le lien symbolique "latest"
cd /opt/monitoring-lab/$USER/reports/latest
ls -la

# Lire un fichier spécifique
cat services.txt
cat app_summary.txt

Exécution automatique via systemd

1. Installation de base (lancement manuel via service)

Installation du service Shell 
# Copier le script dans les binaires système
sudo cp monitor.sh /usr/local/bin/monitor.sh

# Copier le fichier service
sudo cp student-monitor.service /etc/systemd/system/

# Recharger systemd et tester le lancement
sudo systemctl daemon-reload
sudo systemctl start student-monitor.service

2. Automatisation périodique (Le Timer)

Par défaut, le service est de type oneshot (il s'exécute une fois et s'arrête). Pour qu'il se lance automatiquement toutes les heures, il faut créer un Timer.

Créer /etc/systemd/system/student-monitor.timer INI 
[Unit]
Description=Lancer le script de monitoring etudiant toutes les heures

[Timer]
OnCalendar=hourly
Persistent=true

[Install]
WantedBy=timers.target

Ensuite, activez le chronomètre définitivement :

Activer le Timer Shell 
sudo systemctl daemon-reload
sudo systemctl enable --now student-monitor.timer

3. Alternative : Automatisation simple avec Cron

Si vous préférez ne pas utiliser Systemd, vous pouvez utiliser l'outil classique cron.

Éditer la table Cron Shell 
sudo crontab -e

# Ajouter cette ligne à la fin pour une exécution toutes les heures :
0 * * * * /opt/monitoring-lab/maxia/monitor.sh

4. Mise à jour du script

Pour récupérer la version la plus récente du script depuis le dépôt officiel :

Mise à jour Git Shell 
git pull origin main

Automatiser les mises à jour (via Cron) :
Il est possible de configurer `cron` pour télécharger les dernières mises à jour de manière transparente (ex: tous les jours à 2h00).

Planifier la mise à jour (crontab -e) Shell 
# Ajouter cette ligne à la fin du fichier crontab :
0 2 * * * cd /chemin/vers/linux_monitoring_project && git pull origin main
